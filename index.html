You are starting near-zero on DSA — that is completely fine. Senior frontend/RN interviews ask pattern-based problems, not competitive programming tricks. Your goal is to recognise the pattern, not derive algorithms. About 6 weeks of consistent 1-hour daily practice is enough.

Week 1 — Don't code yet. Learn vocabulary.

Foundation first. Zero LeetCode this week.
  • What is Big-O? Learn O(1), O(n), O(n²), O(log n) — just recognise the shapes, don't derive
  • Watch "Big O Notation" by CS Dojo on YouTube (~20 min) — very visual, no maths
  • Mental model: "What do I have → What do I need → What does that suggest?"
  • Arrays and hashmaps (objects) — you use these daily, now learn to think about their cost
⚠️ Do NOT start with hard LeetCode. It will destroy your confidence. The goal week 1 is just vocabulary and mindset.

Week 2 — Arrays & Strings (covers ~40% of frontend DSA)

3 patterns. That's it. Learn them deeply.
  • Two pointers — two indices moving toward each other. Used for: reverse array, find pair with sum, move zeroes
  • Sliding window — a window that moves across the array. Used for: max sum subarray, longest substring
  • Frequency map — count with an object/Map. Used for: anagram check, duplicate finding
// Two Sum — frequency map pattern function twoSum(nums, target) { const seen = {}; for (let i = 0; i < nums.length; i++) { const complement = target - nums[i]; if (seen[complement] !== undefined) return [seen[complement], i]; seen[nums[i]] = i; } } // LeetCode to try: #1 Two Sum, #121 Buy/Sell Stock, #242 Valid Anagram
💡 When stuck after 20 min: look at the solution, understand WHY it works, close it, write it yourself from scratch. This beats grinding 50 problems you don't understand.

Week 3 — Hashmaps & Sets in Depth

Most underrated topic. Turns O(n²) into O(n).
  • Rule of thumb: "I need to look something up fast" OR "I need to count occurrences" → hashmap
  • Set for deduplication and O(1) lookup (vs Array's O(n) includes)
  • One-pass hashmap: store what you've seen so far, check against it as you go
// Contains Duplicate — Set pattern function containsDuplicate(nums) { const seen = new Set(); for (const n of nums) { if (seen.has(n)) return true; seen.add(n); } return false; } // LeetCode: #217 Contains Duplicate, #49 Group Anagrams, #128 Longest Consecutive

Week 4 — Stacks & Basic Recursion

Stacks are everywhere in real code. Recursion feels hard but has one trick.
  • Stack (LIFO) = array with push/pop. Used for: valid parentheses, undo, parsing, "next greater element"
  • Queue (FIFO) = array with push/shift. Used in BFS later
  • Recursion = function calling itself + base case. Think: "what is the smallest version of this problem?"
  • Always draw the call stack on paper for the first 3 recursive problems — it clicks much faster
// Valid Parentheses — classic stack problem function isValid(s) { const stack = []; const map = { ')': '(', '}': '{', ']': '[' }; for (const c of s) { if ('({['.includes(c)) { stack.push(c); } else if (stack.pop() !== map[c]) return false; } return stack.length === 0; } // LeetCode: #20 Valid Parentheses, #155 Min Stack
⚠️ Recursion feels confusing at first — this is completely normal. Trust the base case + return. Don't try to trace the whole call stack in your head; just trust the pattern.

Week 5 — Trees (very common at senior level)

Trees come up in ~30% of senior frontend interviews. Less scary than they sound.
  • A tree node is just: { val, left, right }. That's literally it.
  • DFS (depth-first) = go deep before wide. Use recursion naturally.
  • BFS (breadth-first) = go level by level. Use a queue.
  • For frontend context: trees model the DOM, React component tree, file systems
  • The universal tree pattern: base case → recurse left → recurse right → combine
// Max Depth of Binary Tree — universal pattern function maxDepth(node) { if (!node) return 0; // base case const left = maxDepth(node.left); // recurse left const right = maxDepth(node.right); // recurse right return 1 + Math.max(left, right); // combine } // LeetCode: #104 Max Depth, #226 Invert Binary Tree, #102 Level Order Traversal
💡 When solving tree problems: always ask "what should this function return?" first. Most tree problems are just: do something + recurse left + recurse right.

Week 6 — Sorting, Binary Search & Revision

Understand patterns, not implementations from scratch.
  • You don't need to implement sort — Array.sort() is O(n log n). Know WHEN to sort first.
  • Sorting first can turn an O(n²) problem into O(n log n) — very common trick
  • Binary search: "Is it sorted AND I need to find something fast?" → binary search
  • Revision week: redo 10 previously solved problems without looking at notes. The struggle to recall IS the learning.
// Binary Search — the pattern function binarySearch(arr, target) { let left = 0, right = arr.length - 1; while (left <= right) { const mid = Math.floor((left + right) / 2); if (arr[mid] === target) return mid; else if (arr[mid] < target) left = mid + 1; else right = mid - 1; } return -1; } // LeetCode: #704 Binary Search, #33 Search in Rotated Array, #15 3Sum

Daily practice structure (3–4 hrs/day)

Memory-friendly approach for your study style
  • 30 min — Try to recall yesterday's concept WITHOUT looking at notes first. Write it out. Then check.
  • 45 min — Learn today's concept with one example. Understand it, don't memorise it.
  • 60 min — Solve 1–2 easy problems. If stuck after 20 min → look at solution, rebuild it yourself.
  • 15 min — Write 3 sentences in your OWN words summarising today's learning.
  • Remaining time — if energy exists, try 1 medium problem. If not, stop. Consistency beats intensity.
💡 You WILL forget. That is not failure — that is exactly how memory works. The act of trying to remember before checking is called retrieval practice, and it builds far stronger memory than re-reading. Forgetting + recalling = learning.
Senior frontend/RN system design is NOT "design a distributed database." You'll be asked: Design a news feed. Design a chat feature. Design a multi-step insurance form. Focus is on client architecture, state management, performance, and API design.

The Framework — same structure every single time

Learn this framework first. Use it for every practice session.
  • Step 1 — Clarify (2–3 min): "Is this mobile or web? What scale? Real-time? Offline support needed?"
  • Step 2 — Data model: "What are the main entities? What data does each screen need?"
  • Step 3 — API design: "What endpoints/queries does the client need?"
  • Step 4 — Component architecture: "How do I structure screens and components?"
  • Step 5 — State management: "What's global vs local? Which tool and why?"
  • Step 6 — Edge cases: "Offline? Pagination? Loading/error/empty states?"
⚠️ Never start designing immediately. Spend 2–3 minutes asking clarifying questions. Interviewers reward this — it signals senior-level thinking.

Core Concepts to Know

These appear in almost every frontend system design question.
  • Pagination: Offset-based (?page=2) = simple but breaks on live data. Cursor-based (?after=id) = stable for real-time feeds.
  • Optimistic UI: Update UI before API confirms, rollback on failure. Huge UX win for likes, follows, add-to-cart.
  • Caching: What to cache, where (memory/AsyncStorage/CDN), when to invalidate.
  • WebSocket vs polling: Chat/live updates → WebSocket. Dashboard refresh → polling is fine.
  • Image handling in RN: Lazy load, placeholder → thumbnail → full res, react-native-fast-image for caching.
  • Offline support: Queue actions when offline, sync on reconnect. Critical for mobile health apps.
  • Error states: Every screen needs: loading, success, error, empty. Design all 4.

Week 3 — 5 Systems to Practice Designing

One per day. Time yourself: 30 min each. Draw on paper first.
  • Social media feed — infinite scroll, like/comment, real-time updates
  • Chat application — real-time messages, read receipts, offline message queue
  • E-commerce listing — filters, search, add-to-cart, optimistic UI
  • Photo upload feature — progress indicator, retry logic, background upload
  • Multi-step insurance form — validation, save draft, submit, error recovery (use your Apollo experience here!)
💡 For each design: draw boxes for screens, arrows for data flow on paper FIRST. Then write bullet points for state structure. You don't need to write code in system design — reasoning and trade-offs are what interviewers want to hear.

How to Talk in a System Design Interview

Delivery is as important as content at senior level.
  • Open with: "Before I start designing, let me confirm a few requirements..."
  • Think out loud: "I'm considering X approach because..."
  • Explain trade-offs: "I could use WebSockets or polling here — I'm choosing WebSockets because this is real-time chat where latency matters"
  • Reference real experience: "In the insurance module at Apollo, we solved pagination like this..."
  • Check in mid-design: "Does this approach align with what you're looking for before I continue?"
💡 Your 5 years of production experience at a health-tech company at scale is a MASSIVE advantage. Interviewers respond far better to "I've done this in production" than "I've read about this". Use your real experience constantly.

🚀 React Native 0.76 — New Architecture ON by default (Oct 2024)

The most significant RN release since the framework was created. This is the one every interviewer will ask about.
  • New Architecture is now the default — JSI, Fabric, TurboModules enabled in all new projects. No more opting in.
  • React Native DevTools — single unified debugging stack (replaces Flipper as default). Zero setup. Opens via Dev Menu or pressing j in CLI. Includes React DevTools built-in.
  • Metro ~15x faster resolver — warm builds ~4x faster. Cold start significantly improved.
  • New style props — boxShadow and filter (New Architecture only). Brings RN styling closer to CSS.
  • Android ships as single .aar library — smaller APK size, faster Android startup.
  • Full React 18 support — Suspense, useTransition, automatic batching now work correctly in RN via Fabric.
💡 Interview angle: "0.76 is the moment the New Architecture graduated from experimental to production-ready default. If you're starting a new project today, you're on the New Architecture automatically. The main impact I care about is that Fabric enables React 18 concurrent features properly, and Metro's resolver improvement makes day-to-day development noticeably faster."
📖 Official 0.76 release blog  ·  New Architecture docs

🎨 React Native 0.77 — CSS features, Swift default, Metro cleanup (Jan 2025)

Focus on web alignment, better styling, and modernising the iOS template.
  • New CSS properties (New Architecture only) — display: contents, boxSizing (content-box / border-box), mixBlendMode, outline* properties. Big step toward web/RN parity.
  • Swift as default iOS template — new projects use Swift AppDelegate instead of Objective-C. Objective-C still supported but no longer default.
  • Metro log forwarding removed — console.log no longer streams through Metro terminal. Use React Native DevTools Console panel instead (Chrome DevTools Protocol).
  • Android 15 + 16KB memory page support — edge-to-edge UI compatibility, improved memory handling on newer Android devices.
  • react-native init deprecation continues — official guidance is now npx create-expo-app or community CLI.
  • ScrollView sticky headers + absolute positioning fixes — layout consistency improvements.
⚠️ Gotcha to know: If you upgraded to 0.77 and use Expo Updates for OTA, the :hermes_enabled flag in Podfile is still required even though it looks deprecated. Removing it breaks OTA updates.
💡 Interview angle: "0.77 shows Meta's clear goal of aligning RN with web CSS. The new styling props — especially boxSizing and display:contents — let you write layout code that's mentally identical to web CSS. The console.log change is a breaking DX change: Metro no longer streams logs, so the team needs to move to React Native DevTools."
📖 Official 0.77 release blog

⚛️ React Native 0.78 — React 19 support (Feb 2025)

React 19 lands in React Native. Major new hooks and patterns available.
  • React 19 fully supported — the biggest headline. New hooks and patterns available in RN.
  • useActionState — manage async action state (form submissions, mutations). Replaces manual loading/error useState pairs.
  • useOptimistic — built-in optimistic UI updates during async requests. No more manual rollback logic.
  • use() — read Promises or Context directly during render. Suspends until resolved.
  • ref as prop — pass ref as a regular prop without forwardRef. forwardRef still works but no longer required.
  • propTypes removed — deprecated APIs from React 18 are gone. Update any libraries still using propTypes.
  • Android XML drawables — native support for Android vector drawables in the Image component. No extra libraries needed.
  • React Compiler support — simplified opt-in setup for the React Compiler (auto-memoization).
💡 Interview angle: "0.78 brings React 19 to mobile, which means useOptimistic is now available natively — no more writing rollback logic by hand for optimistic UI. useActionState simplifies form mutation patterns significantly. The ref-as-prop change is also worth knowing: forwardRef is no longer needed for passing refs to custom components."
📖 Official 0.78 release blog  ·  React 19 full changelog

⚡ React Native 0.79 — Metro 0.82, faster Android startup (Apr 2025)

Performance and tooling focused release.
  • Metro 0.82 — 3x faster cold startup for the bundler. Package exports support finally lands (resolves many third-party library issues).
  • Faster Android app startup — JS bundle compression skipped on Android, improving cold start time meaningfully.
  • Native module registration changes — improvements to how native modules are registered for TurboModule compatibility.
📖 Official 0.79 release blog

🔒 React Native 0.80–0.82 — Legacy Architecture frozen (2025)

The old bridge-based architecture is officially end-of-life.
  • 0.80 — React 19.1 support, JS API changes, Legacy Architecture frozen (no new features, security fixes only).
  • 0.82 — Old architecture permanently disabled. newArchEnabled=false no longer works. All apps must be on New Architecture.
  • Reanimated 4 stable — CSS animations and transitions via Reanimated. Worklets extracted into a standalone package.
  • FlashList rewritten — Shopify rewrote FlashList from the ground up for better New Architecture compatibility.
  • Legend List 1.0 — new JS-only virtualized list for New Architecture. No native code. Uses synchronous layout APIs.
  • Hermes v1 (0.84) — first stable major version of Hermes. WebAssembly support added.
⚠️ Production reality: As of 0.82, you cannot fall back to the old bridge. Any third-party library that hasn't migrated to New Architecture will break. Always check library compatibility before upgrading.
💡 Interview angle: "0.82 is a hard cutoff — the old bridge is gone. This is the most impactful production concern for teams upgrading today. Every native module dependency needs a New Architecture compatible version, or you need to fork and migrate it yourself."
📖 Official 0.80 release blog  ·  All releases on GitHub

📦 Ecosystem changes worth knowing

Libraries and tooling that interviewers at product companies now expect you to know about.
  • Expo is now the recommended way to start — react-native init is deprecated. npx create-expo-app is the official starting point even for bare workflow projects.
  • React Native DevTools — replaces Flipper as the default debugger. Built on Chrome DevTools Protocol. No setup required.
  • Reanimated 4 — CSS animations and transitions. Worklets are now a standalone package usable without Reanimated.
  • FlashList v2 — rewritten for New Architecture. Still the recommended replacement for FlatList on large lists.
  • Nitro Modules — new alternative to TurboModules for native module authoring. Faster, simpler spec. Gaining adoption in 2025.
  • Re.Pack 5.0 — alternative to Metro bundler with Module Federation support. For micro-frontend RN setups.
  • Unistyles 3.0 — styling library built in C++. Surgical style updates, eliminates unnecessary re-renders.

General Story Delivery Framework (STAR-T)

Use this structure for every feature story — keep it to ~2 minutes
  • S — Situation: What was the context? What problem existed?
  • T — Task: What was YOUR specific responsibility?
  • A — Action: What did you do? (most time here — your decisions and why)
  • R — Result: What was the outcome?
  • T — Technical depth: 1-2 sentences on the hardest part — invite them to go deeper
⚠️ Practice each story out loud at least 3 times before the interview. Reading ≠ remembering. The slight awkwardness of saying it aloud is exactly what builds fluency.
